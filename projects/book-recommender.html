Memorabilia — Book Recommendation System

A personalized recommendation engine for a reading app, built to answer a simple question: will actually relevant picks get people reading more than a generic bestseller list?

Python TensorFlow/Keras Pandas Flask
Live Demo GitHub Repository

The challenge

Business context

Memorabilia's reading app had engagement problems familiar to any content platform: readers opened the app, saw a generic bestseller list, and often didn't come back. The team wanted personalized recommendations that could lift both reading engagement and, longer term, literacy — but with real interaction data that was sparse for most users.

Questions I asked

📖

Do personalized picks outperform a bestseller list?

🕳️

How sparse is the interaction data, really?

🎭

Which genres are hardest to recommend confidently?

01 / EDA JOURNEY

Just how sparse is the ratings matrix?

Before picking a modeling approach, I checked how many books the typical user had actually rated — this determines whether collaborative filtering has enough signal to work at all.

💡 Most users had rated fewer than 5 books — the classic cold-start problem collaborative filtering needs to be designed around, not ignored.

02 / FEATURE ENGINEERING & MODELING

Neural Collaborative Filtering

I modeled user and book IDs as learned embeddings, feeding their interaction into a small neural network (Neural Collaborative Filtering) rather than a traditional matrix factorization — better able to capture non-linear preference patterns as the embedding size grew.

💡 Validation loss plateaued around epoch 12 — training past that point mostly overfit to the power users with lots of ratings.

Did it actually work?

Engagement vs. the bestseller baseline

The real test wasn't offline accuracy — it was whether people opened what the model recommended more than they opened the generic list it replaced.

Key takeaways

What the data actually said

📚

NCF-personalized recommendations meaningfully outperformed the generic bestseller list on click-through.

🕳️

New users with little history remained the hardest to serve well — the model needs a fallback for them.

⚙️

Deploying behind a Flask API on GCP kept inference fast enough for in-app use without a heavy backend.

So what should change

Business recommendations

01

Add a content-based fallback (genre, author, tags) for new users until they've rated enough books for collaborative filtering to kick in.

02

Instrument click-through and completion tracking in the app so the model can be retrained on real engagement signal, not just star ratings.

03

Expand book metadata (author, tags, reading level) to give the model more signal for genres where confidence was lowest.

Reflection

The sparsity check early on saved me from over-investing in a pure collaborative-filtering approach — I designed the fallback path from the start instead of bolting it on later. If I revisited this, I'd prioritize instrumenting engagement tracking before the model launch, not after, so the retraining loop could start immediately.
